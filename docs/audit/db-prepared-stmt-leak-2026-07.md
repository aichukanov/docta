# Инцидент: утечка prepared statements в MySQL — июль 2026

## Симптом

Периодически переставали работать страницы, зависящие от БД: не открывалась деталь лекарства
(`/medicines/sinacilin-250-mg5ml`), не работал поиск (`/medicines?name=SINACILIN`) — и так по всему сайту.
Проблема накапливалась за несколько дней и лечилась **только перезапуском pm2**. Триггер по времени
совпал с появлением странных ботов: невысокий объём (50–100/час, 1–2k/день), сервер они почти не грузили,
но через несколько дней всё падало.

Диагностика — по выгрузке pm2-логов (`docta-error-4..7.log`, ~788 КБ каждый).

## Что в логах

|---|---|---|

| `connect ECONNREFUSED 127.0.0.1:3306` | ~217 | Следствие: перезапуски MySQL/pm2 для лечения. |

| `Cannot find module .../.output/server/chunks`, `ENOENT` | единицы | Транзиентное во время деплоя/рестарта. |

## Корневая причина

mysql2 держит LRU-кэш подготовленных выражений **на каждое физическое соединение**, дефолт
`maxPreparedStatements: 16000` (см. `node_modules/mysql2/lib/connection_config.js:194`). При
`connectionLimit: 10` теоретический потолок пула — до **160 000** серверных выражений, тогда как
серверный `max_prepared_stmt_count` = **16382**.

Наши списки и детали генерируют огромное количество **уникальных строк SQL**:

- динамический WHERE (разные комбинации фильтров);
- разное число плейсхолдеров `?` в `IN (...)`;
- **главное** — `LIMIT ${pageSize} OFFSET ${offset}` вшит прямо в текст запроса, поэтому
  **каждая страница пагинации = новая строка SQL = новый prepared statement**. Множитель безграничный.

Боты, обходящие множество разных пагинированных/фильтрованных URL, за несколько дней забивают
LRU-кэши всех соединений разными выражениями. Глобальный счётчик переваливает за 16382 → сервер
отказывает в создании новых statement → **падают ВСЕ запросы** (детали, поиск, списки), пока соединения
не сбросятся. Перезапуск pm2 закрывает соединения вместе с их выражениями — отсюда «лечится только
рестартом». Это **утечка/накопление, а не нагрузка** — потому и низкий траффик её вызывал.

## Фикс (применён)

`server/common/db-mysql.ts` — в `createPool` добавлено `maxPreparedStatements: 200`:

```
10 соединений × 200 = 2000 statements « 16382
```

LRU закрывает старые выражения при вытеснении (COM_STMT_CLOSE, см.
`node_modules/mysql2/lib/base/connection.js:83-88`), поэтому глобальный счётчик держится ограниченным
с большим запасом на прочие соединения. 200 «горячих» выражений на соединение с запасом покрывают
реальный набор запросов.

### Проверка (эмпирически, локально)

Прогнал 300 уникальных пагинированных запросов (`/api/medicines/list` и `/api/clinics/list`,
page 1..150) — `SHOW GLOBAL STATUS LIKE 'Prepared_stmt_count'` вышел на **плато 200** и перестал расти.
До фикса те же 300 запросов копили бы 300+ выражений и продолжали расти к 16382. Обе исходные ссылки
(`/medicines/sinacilin-250-mg5ml` и `?name=SINACILIN`) отдают 200/результаты.

**Мониторинг после деплоя:** `SHOW GLOBAL STATUS LIKE 'Prepared_stmt_count';` должен оставаться низким
(двузначные–низкие сотни), а не расти день ото дня.

## Побочный фикс: поиск лекарств по названию с символами

`common/validation.ts` — `validateName` допускал только буквы/цифры/пробелы/`.,-`. Реальные названия
лекарств содержат `/ + ( ) : % ’ ® ™` (например `SINACILIN®`, `SAKUBITRIL/VALSARTAN`,
`ASPIRIN® PLUS C 400mg/240mg`). При таком символе валидация падала, и эндпоинт **молча выбрасывал
фильтр по имени, отдавая весь каталог** (2523 позиции) — для пользователя это «поиск не работает».

Исправлено:
- расширен допустимый набор символов в `validateName` (поиск идёт через bound-параметры `LIKE ?`,
  так что это санитарная проверка, не защита от инъекций);
- `server/api/medicines/list.ts` и `server/api/medications/list.ts`: при невалидном имени теперь
  возвращается пустой результат (`1 = 0`), а не полный каталог — молчаливый сброс фильтра был ловушкой.

## Требует внимания (кодом не трогал)

**`Unknown column 'r.status' / 'c_rank.status' / 'c.status'`** — источники:
`server/api/clinics/list.ts:328`, `server/api/labtests/details.ts`, `server/api/services/details.ts`,
фоновый `[rank-score]`. Локально колонки `clinics.status` и `reviews.status` существуют. Ошибки идут
кластером (не до конца логов) → похоже на окно **до применения миграций на prod**. Проверить, что
миграции каталога клиник/отзывов (004/005/006) накатаны на проде.

## Возможный follow-up (оценён, решено НЕ делать сейчас)

Убрать сам churn подготовки/закрытия выражений. Аптайм уже спасён кэш-лимитом при нулевом риске;
выигрыш от follow-up — невидимый (MySQL на том же хосте, round-trip доли мс, траффик 1–2k/день).

| Вариант | Охват | Сложность | Эффект |
|---|---|---|---|
| A. Параметризовать LIMIT/OFFSET | 12 файлов с инлайновым `LIMIT ${…}` | Low-Medium | Убирает безграничный множитель «номер страницы»; WHERE/IN variety остаётся |
| B. Перевести read-эндпоинты на `.query()` | ~10–15 файлов чтения | Medium | Полностью убирает класс проблемы; смена семантики (escaping vs binding), каждый запрос ревьюить |

**Подводный камень варианта A** (проверено на mysql2 3.22 + текущий MySQL): `LIMIT ? OFFSET ?` через
`.execute()` с **числами** кидает `ER_WRONG_ARGUMENTS`, работает только со **строками**
(`['5','10']`). Легко забыть → нужен тест на каждый эндпоинт.

**Рекомендация:** сейчас не делать. Если понадобится defense-in-depth — брать только вариант A
по list-эндпоинтам (лучший value/effort). Вариант B — оверкилл, пока кэш ограничен.
