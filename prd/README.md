# Product Requirements Documents (PRD)

Эта папка содержит детальные PRD документы для функциональности проекта docta.me.

## Структура

Каждый PRD организован в виде модульной структуры:

- **Главный файл** (`index.md`) - навигация и обзор
- **Секции** - отдельные файлы для каждой части документа
- **Итерации** - отдельная папка с детальным описанием каждой итерации разработки

Каждая итерация:

- Может быть выполнена LLM автономно
- Требует ручной проверки перед переходом к следующей итерации
- Содержит четкие критерии приемки (acceptance criteria)
- Включает технические детали реализации

## Формат PRD

Каждый PRD использует модульную структуру с папкой, содержащей `index.md` и отдельные файлы секций.

Проект разделен на несколько независимых PRD для лучшей управляемости и возможности параллельной разработки.

## Создание нового PRD

### ⭐ Рекомендуемый подход: Модульная структура (как auth/)

**Создавайте новые PRD по аналогии с [auth/](auth/)** - используйте модульную структуру с папками и отдельными файлами.

#### Шаг 1: Создайте структуру папок

```
prd/
└── <название-фичи>/              # Например: notifications/, payments/, reviews/
    ├── index.md                   # Главная страница с навигацией
    ├── PROGRESS.md                # 📊 Отслеживание прогресса и статусов
    ├── 01-overview.md             # Обзор, цель, контекст
    ├── 02-requirements.md         # Функциональные и нефункциональные требования
    ├── 03-architecture.md         # Стек, API endpoints, компоненты, диаграммы
    ├── 04-database.md             # Схема БД, миграции (если применимо)
    ├── 05-risks-and-metrics.md    # Риски, метрики успеха, вопросы
    └── iterations/                # Папка с итерациями разработки
        ├── README.md              # Обзор всех итераций с таблицей статусов
        ├── iteration-01-*.md      # Итерация 1
        ├── iteration-02-*.md      # Итерация 2
        └── ...                    # И так далее
```

#### Шаг 2: Скопируйте шаблон из auth/

```bash
# Создайте новую папку
mkdir prd/<название-фичи>

# Скопируйте структуру из reviews/ как основу
cp -r prd/reviews/* prd/<название-фичи>/

# Адаптируйте содержимое под вашу фичу
```

#### Шаг 3: Заполните секции

1. **index.md** - создайте навигацию по всем секциям и итерациям
2. **PROGRESS.md** - создайте файл для отслеживания прогресса (см. [reviews/PROGRESS.md](reviews/PROGRESS.md))
3. **01-overview.md** - опишите цель, контекст, предположения
4. **02-requirements.md** - перечислите FR и NFR, определите scope
5. **03-architecture.md** - опишите API, компоненты, диаграммы
6. **04-database.md** - если нужны изменения БД, опишите схему
7. **05-risks-and-metrics.md** - оцените риски, определите метрики
8. **iterations/** - разбейте работу на логические итерации

#### Шаг 4: Структурируйте итерации

Каждая итерация должна содержать:

- ✅ Цель и зависимости
- ✅ Список задач
- ✅ Технические детали (примеры кода, файлы)
- ✅ Критерии приемки (AC-1, AC-2, ...)
- ✅ Инструкции по проверке
- ✅ Статус (Not Started / In Progress / Review / Completed)

### 📚 Референс: Пример реальной структуры

Смотрите [reviews/](reviews/) как эталон правильной структуры:

- ✅ Чистая навигация с breadcrumbs
- ✅ Логическое разделение по секциям
- ✅ Детальные итерации с примерами кода
- ✅ Относительные ссылки между файлами

### 🎯 Преимущества модульной структуры

✅ **Навигация** - легко найти нужную секцию  
✅ **Модульность** - независимое редактирование секций  
✅ **Git** - понятные diff'ы, легче разрешать конфликты  
✅ **Масштабируемость** - легко добавлять итерации  
✅ **Читаемость** - меньше когнитивной нагрузки  
✅ **Совместная работа** - несколько человек могут работать параллельно
