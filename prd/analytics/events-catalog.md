# Каталог событий аналитики

**Обновлено:** 2026-06-12 (фаза Б включена)
**Источник истины для кода:** `types/analytics.ts` (типы) + `composables/use-analytics.ts` (отправка)

Все события уходят в Mixpanel (EU-кластер, `api-eu.mixpanel.com`) только после
согласия на cookies. Имена событий и свойств — snake_case. PII (телефоны,
email, адреса) в свойства не кладётся — только тип контакта и ссылка на
сущность.

---

## Автоматические свойства

К каждому событию `trackEvent` добавляет:

| Свойство    | Значение                                                                |
| ----------- | ----------------------------------------------------------------------- |
| `page_type` | Тип страницы по имени роута: `home`, `clinic_list`, `clinic_detail`, `doctor_detail`, `service_detail`, `labtest_detail`, `medicine_detail`, `medication_detail`, `article`, … (см. `getPageType`) |

Super properties (регистрируются при инициализации и на каждом переходе,
попадают во **все** события, включая pageview):

| Свойство    | Значение                                  |
| ----------- | ----------------------------------------- |
| `locale`    | Текущая локаль интерфейса (`ru`, `en`, …) |
| `page_type` | Тип текущей страницы                      |

Mixpanel сам добавляет `$current_url`, `$referrer`, устройство, гео и т.д.

### Просмотры страниц

- Полная загрузка страницы трекается автоматически (`track_pageview: true`
  в init) — событие `$mp_web_page_view`.
- Клиентские SPA-переходы трекаются вручную в `router.afterEach`
  (`mixpanel.track_pageview()`), super properties обновляются до отправки.
- Просмотры **листингов** покрыты pageview: применённые фильтры видны в
  query-параметрах `$current_url`. Отдельного события listing-просмотра нет.

## Ссылка на сущность (`entity_*`)

У событий, связанных с сущностью каталога, всегда один и тот же набор:

| Свойство      | Тип                                                                | Описание                            |
| ------------- | ------------------------------------------------------------------ | ----------------------------------- |
| `entity_type` | `clinic \| doctor \| service \| labtest \| medicine \| medication \| article` | Тип сущности                        |
| `entity_id`   | number (у статических статей — slug-строка)                        | id из БД                            |
| `entity_slug` | string                                                             | slug для URL                        |

`/medications/*` (аптечный прайс клиник) — `entity_type: 'medication'`,
`/medicines/*` (реестр лекарств) — `entity_type: 'medicine'`: это разные
таблицы (medications и med_medicines), их id пересекаются.

## События

### `entity_viewed`

Просмотр детальной страницы сущности. Шлётся при заходе и при клиентском
переходе между сущностями одного типа (компонент страницы переиспользуется).

| Свойство        | Обяз. | Описание                                          |
| --------------- | ----- | ------------------------------------------------- |
| `entity_*`      | да    | См. выше                                          |
| `entity_name`   | нет   | Каноническое (нелокализованное) название          |
| `clinics_count` | нет   | Сколько клиник предлагают услугу/анализ/лекарство |

Где шлётся: детальные страницы клиники, врача, услуги, анализа, лекарства
(оба раздела), 2 статические статьи.

### `entity_link_clicked`

Клик по ссылке/карточке сущности — переходы между сущностями
(service→clinic, doctor→clinic, clinic→doctor, medicine→clinic, article→…).
Источник перехода определяется по `page_type` события.

| Свойство      | Обяз. | Описание                          |
| ------------- | ----- | --------------------------------- |
| `entity_*`    | да    | **Целевая** сущность              |
| `entity_name` | нет   | Название целевой сущности         |
| `position`    | нет   | Позиция карточки в списке (с 1)   |

Где шлётся: карточка клиники (`clinic/summary-header.vue`), карточка
врача (`doctor/info.vue`), карточка/строка услуги-анализа-лекарства
(`priced-item-card.vue`, `service-info.vue`), заголовок и «показать все
клиники» в `list-card.vue`, имя клиники в попапе карты, ссылки на
врачей/клиники в статьях.

### `contact_clicked`

Клик по контакту (tel:, мессенджер, сайт, соцсеть). Email сейчас не
кликабелен в UI (только копирование). Сущность-владелец контакта берётся из
provide/inject-контекста (детальные страницы, карточка клиники, попап карты)
и в редких случаях может отсутствовать.

| Свойство       | Обяз. | Описание                                                                     |
| -------------- | ----- | ---------------------------------------------------------------------------- |
| `contact_type` | да    | `phone \| email \| address \| website \| whatsapp \| viber \| telegram \| facebook \| instagram` |
| `entity_*`     | нет   | Владелец контакта (клиника/врач)                                             |

«Переход на сайт клиники» = `contact_clicked` с `contact_type: 'website'`.

### `contact_copied`

Копирование контакта кнопкой-копиркой. Раньше событие слало само значение
(телефон/email) — это PII, заменено на тип контакта + сущность.

| Свойство       | Обяз. | Описание                  |
| -------------- | ----- | ------------------------- |
| `contact_type` | да    | Как в `contact_clicked`   |
| `entity_*`     | нет   | Владелец контакта         |

Примечание: отдельной кнопки «скопировать адрес» в UI нет — `address`
зарезервирован в типах на будущее.

### `search_performed`

Поиск по названию на листинге (debounced, не каждый кейстрок).

| Свойство        | Обяз. | Описание                       |
| --------------- | ----- | ------------------------------ |
| `query`         | да    | Поисковая строка               |
| `results_count` | нет   | Кол-во результатов после поиска |

### `filter_applied`

Применение фильтра на листинге.

| Свойство        | Обяз. | Описание                                |
| --------------- | ----- | --------------------------------------- |
| `filter_name`   | да    | Поле стора фильтров: `cityIds`, `languageIds`, `specialtyIds`, `clinicTypeIds`, `clinicIds`, `categoryIds`, `serviceCategoryIds`, `dispensingModeIds`, `atcGroupIds`, `substanceIds`, `pharmaFormIds`, `manufacturerIds`, `openNow`, `minRating` |
| `filter_value`  | да    | Применённое значение                    |
| `results_count` | нет   | Не отправляется сейчас (refetch асинхронный) |

Реализация: `composables/use-filter-tracking.ts` следит за неймспейсом
стора фильтров на каждой листинг-странице; первичная синхронизация из URL
не трекается.

### `filter_cleared`

Снятие фильтров: поле(я) фильтра стали пустыми. `filters_count` — сколько
полей сброшено этим действием. Отдельной кнопки «сбросить всё» в UI нет.

### `map_opened`

Карта смонтирована (она грузится лениво — по скроллу до карты, переключателю
«карта» на листинге клиник или кнопке «показать на карте»).
`markers_count` — сколько клиник на карте.

### `map_marker_clicked`

Клик по маркеру клиники на карте; `entity_*` — клиника маркера. Клик по
названию клиники в попапе — отдельное `entity_link_clicked`.

## User identification

- **Login:** `mixpanel.identify(userId)` + user property `auth_provider`
  (`google | telegram | facebook | password`). Срабатывает в
  `plugins/analytics-user.client.ts` при появлении пользователя в сторе
  (логин, регистрация, OAuth-redirect, восстановление сессии) — и только
  при наличии consent.
- **Logout:** `mixpanel.reset()` — отвязывает устройство от профиля.
- Имя/email в Mixpanel **не** отправляются (PII).

## Как проверить (debug mode)

1. `npm run dev` — в dev-режиме Mixpanel инициализируется с `debug: true`,
   события логируются в консоль браузера (`[Mixpanel] …`).
2. Принять cookies (баннер).
3. Открыть детальную страницу клиники → в консоли `entity_viewed`.
4. Кликнуть телефон/скопировать контакт → `contact_clicked` / `contact_copied`
   без значения телефона в props.
5. Залогиниться → в консоли identify; в Mixpanel Live View события поедут
   с `$user_id`.
6. Прод: Mixpanel → Events (Live View) — фильтр по своему `$user_id` или
   городу.
