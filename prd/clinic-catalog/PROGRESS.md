## Sitemap для фильтров каталога (решение 2026-06-12)

Фильтрованные страницы — самостоятельные SEO-лендинги (canonical включает query).
Что включаем в sitemap (`server/common/sitemap/filters/clinics.ts`):

- ✅ **Город** (`?cityIds=N`) — было; переведено с перебора enum на реальные
  города с опубликованными клиниками.
- ✅ **Тип клиники** (`?clinicTypeIds=N`) и **тип×город** — добавлено: самый
  частотный класс запросов («стоматолошка ординација Подгорица», «dental clinic
  Budva»), стабильный состав, готовые тайтлы `ClinicsType[City]`. Порог для
  тип×город — ≥2 опубликованных клиник (`SITEMAP_CLINIC_TYPE_CITY_MIN_CLINICS`).
- ❌ **Рейтинг** — нет спроса («клиники с рейтингом 4+» никто не ищет),
  нестабильный состав (каждый отзыв/модерация меняет страницу), thin-дубль
  базового списка ×3 порога ×6 локалей.
- ❌ **«Открыто сейчас»** — контент зависит от времени суток.
- ❌ **Специализация клиник** — интент «кардиолог Подгорица» уже занят
  страницами `/doctors?specialtyIds=X&cityIds=Y` из sitemap; вторая поверхность
  под тот же запрос — каннибализация. Вернуться, только если решим, что интент
  «клиника с кардиологией» ≠ «врач-кардиолог».

## Сортировки: расстояние vs цена vs рейтинг (2026-06-12)

### Проблема

После того как геолокация стала глобальной (селектор в хедере), три сигнала
ранжирования конфликтуют:

1. **Расстояние затирает rank_score.** В каталоге `/clinics` при известной
   локации `ORDER BY distance ASC, rank_score DESC` — расстояние полностью
   перекрывает общий рейтинг (`server/utils/entity-ranking.ts`: байесовский
   рейтинг 0.40 + объём отзывов 0.25 + **заполненность профиля 0.20** +
   свежесть 0.10 + ответы 0.05). Пустой профиль в 500 м обгоняет заполненную
   клинику с отзывами в 2 км — качество и доверие перестают влиять на порядок.
2. **Цена игнорирует и расстояние, и рейтинг.** На детальных страницах
   услуг/анализов/лекарств клиники идут строго от дешёвой к дорогой
   (`getPriceOrderBySQL`): первой может стоять дешёвая клиника в 150 км,
   расстояние на карточках вообще не показывается.
3. Подсказка в хедере обещает «город используется только для сортировки по
   расстоянию» — фактически distance-сортировка есть только на `/clinics`.

### Карта сортировок (до изменений)

| Страница | Порядок клиник | Где сортируется |
| --- | --- | --- |
| `/clinics` | локация известна и нет фильтра города → расстояние ASC (rank_score — tie-breaker); иначе rank_score DESC | SQL, перезапрос с клиента |
| `/services`, `/labtests`, `/medications` (листинги) | внутри карточки — цена ASC, без цены в конце (видны первые 3) | SQL, SSR |
| `/services/[slug]` + аналоги | цена ASC, без цены в конце | SQL (GROUP_CONCAT), SSR |
| `/doctors`, `/medicines` | rank_score / реестр без цен клиник | не затронуты |

### Решение (выбор пользователя, 2026-06-12)

**Единый механизм ранжирования** вместо трёх несвязанных порядков: везде, где
выводится список клиник, сортировать по композитному скору. Расстояние —
компонент рейтинга, а не замена ему; цена остаётся видимым бейджем.
Явный переключатель сортировки не вводим.

```
composite = rank_score                                  // 0..1, общий рейтинг
          + 0.5 × 2^(−км/10)                            // вклад близости
          + 0.1 (если у клиники указана цена позиции)   // только услуги/анализы/лекарства
```

Рассматривались варианты: чисто ценовая сортировка + фильтр города (отклонено —
«мало, сортировать по цене»), явный переключатель цена/расстояние (отклонено —
лишний контрол), бакеты расстояния (отклонено — жёсткие границы),
мультипликативное затухание (отклонено — убивает межгородское сравнение
качества). Выбрана **аддитивная экспонента**: 0 км → +0.5, 10 км → +0.25,
25 км → +0.09, 100 км → ~0 — между далёкими городами решает качество, внутри
города близость даёт ощутимое преимущество. Роль цены — **мягкий бонус +0.1**
за указанную цену (вместо прежней жёсткой группы «без цены — в конец»).

### Реализация

- `common/ranking.ts` — константы и формула (TS, клиент);
  `server/common/utils.ts` → `getClinicRankOrderBySQL()` (заменил
  `getPriceOrderBySQL`) и `server/api/clinics/list.ts` — SQL-зеркала.
- `/clinics`: ORDER BY = композит (rank_score + близость) при известной
  локации; перезапрос с координатами с клиента, как раньше. Спецслучай FR-2.6
  («фильтр города отключает сортировку») удалён: у клиник далёкого
  отфильтрованного города вклад близости ≈ 0 у всех, порядок сам вырождается
  в rank_score.
- Детальные `/services|labtests|medications/[slug]` и карточки листингов:
  SSR-порядок из SQL = rank_score + бонус за цену (без локации); на клиенте
  `composables/use-clinic-ranking.ts` пересортировывает с учётом расстояния
  (rank_score клиник теперь отдаётся в `/api/clinics/list` как `rankScore`).
  На карточках клиник появилось расстояние «· 2,3 км».
  В карточках листингов состав первых 3 клиник фиксирует сервер (цены обрезаны
  до этого же числа) — клиент только переупорядочивает видимые.
- Подсказка хедера переформулирована: «город влияет на порядок — ближние выше,
  с учётом общего рейтинга; это не фильтр» (6 локалей).
- Аналитика не добавлялась: переключателя сортировки нет, действий
  пользователя не появилось.

### Инварианты (не ломать)

- SSR/SEO: серверная выдача не зависит от локации. Без координат композит
  вырождается в `rank_score + бонус за цену` — это и есть SSR-порядок,
  клиентская пересортировка стартует с него (гидрация не прыгает).
- Локация ≠ фильтр: все клиники остаются в выдаче.

## Что проверить руками

1. **Геолокация:** инкогнито → `/clinics` → с черногорского IP город
   определился, список пересортирован, на карточках расстояние. На локальном /
   не-черногорском IP город НЕ определён (хедер показывает «Город»), порядок —
   чистый rank_score, расстояний нет.
2. **Ручной выбор города:** селектор «Ваше местоположение» → выбрать Bar →
   пересортировка; F5 — выбор сохранён (localStorage); «Определять
   автоматически» — сброс.
3. **Авторизованный:** выбрать город → проверить `auth_users.preferred_city_id`
   (после миграции) → перелогин — город из БД.
4. **Фильтр по городу:** включить чужой город — порядок внутри него по
   rank_score (вклад близости ≈ 0 у всех); включить свой — ближние выше;
   расстояние на карточках остаётся в обоих случаях.
5. **Рейтинг/специализация:** фильтры комбинируются, URL обновляется,
   `<title>` получает суффиксы.
6. **Карта (мобильный/узкое окно ≤950px):** переключатель «Карта» у заголовка,
   кластеры разлетаются при зуме, клик по маркеру — попап со ссылкой на
   клинику; фильтры сужают набор маркеров; карта фулскрин, кнопка «К списку»
   возвращает; URL не меняется, F5 всегда открывает список.
7. ipapi.co заблокировать (DevTools → Network request blocking) → город не
   определён, список по rank_score, страница работает.
