## Что реализовано (итерации 2–6, 2026-06-11)

### База данных

Миграция `server/sql/migrations/006-clinic-cabinet.sql` (см. APPLIED.md):

- `clinics.created_by INT NULL` — FK на `auth_users(id)` ON DELETE SET NULL
- `clinics.status ENUM('draft','pending_verification','published','rejected') NOT NULL DEFAULT 'published'`
- индексы `idx_clinics_status`, `idx_clinics_created_by`
- `logo_url` уже существовал — в миграцию не входит

Решение по статусам (итерация 6): пользовательский флоу — `draft ↔ published`
(публикация без модерации, скрытие возвращает в draft).
`pending_verification` / `rejected` зарезервированы под PRD clinic-verification;
бейдж доверия — отдельное поле `is_approved`.

### API (по образцу doctors/my-profile)

- `GET /api/clinics/my-list` — клиники текущего пользователя (у пользователя их может быть несколько)
- `POST /api/clinics/my-create` — создание клиники (status='draft', created_by=user)
- `POST /api/clinics/my-update` — обновление (владелец или админ); слаг следует за названием с redirect'ом
- `POST /api/clinics/my-set-status` — publish/hide; публикация требует название+адрес+координаты (`CLINIC_INCOMPLETE`)
- Общие хелперы: `server/common/clinic-cabinet.ts` (валидация, ownership, sync M:N, upsert графика)
- Коды: `CLINIC_*` в `server/utils/api-codes.ts`

### Видимость черновиков

- `details.ts`: не-published клиника → null (404) для всех, кроме владельца/админа; в ответе `status`, `isOwner`
- `list.ts`: `WHERE c.status='published'` (включая стор клиник и селекты)
- `reviews.ts`: только published
- sitemap: `filters/clinics.ts`, `filters/clinic-subpages.ts`, запрос reviews-страниц — только published
- Публичная страница: owner-баннер (`components/clinic/owner-banner.vue`) + плашка «не опубликована», `noindex` для не-published
- Интеграционный проход 2026-06-12: гейтинг добавлен также в `clinics/items-summary.ts`
  (подстраницы клиники) и `reviews/create.post.ts` (черновики не принимают отзывы)

**Известное ограничение (вернуться при добавлении управления услугами из кабинета):**
списки/детали услуг, анализов и лекарств агрегируют `clinic_*`-связки без проверки
`clinics.status` (только ID и цены). Сейчас это безопасно: карточки клиник резолвятся
на фронте из стора, который ходит в gated `/api/clinics/list`, а привязать
услуги к draft-клинике из кабинета нельзя. Когда появится управление услугами —
добавить `status='published'` в join'ы `services|labtests|medications` list/details
и sitemap `filters/services|labtests|doctors`.

### UI (зона components/profile/**)

- `tab/clinics.vue` — список клиник со статусами, создание, редактирование (заменил заглушку)
- `ClinicCard.vue`, `ClinicStatusBlock.vue`, `ClinicsEmptyState.vue`
- `ClinicEditForm.vue` — название (SR/SR-CYRL/RU), типы, город, адрес/район (лок.), индекс,

  график работы, markdown-описание (6 локалей), логотип (upload в /uploads/clinics)
- `ClinicMapPicker.vue` — клик по карте ставит точку (`use-leaflet` + новый `onMapClick`)
- `ClinicWorkingHoursEditor.vue` — локализованный редактор графика (v-model)
- `i18n/clinic-profile.ts` — 6 локалей
- `pages/profile.vue` — снята пометка «скоро» с таба клиник
