## 1. Проблема и идея

Аудитория docta.me — русскоязычные (и др.) туристы/переселенцы в Черногории.
Человек знает лекарство по домашнему бренду (Нурофен, Смекта, Энтеросгель) и не
понимает, что купить в местной аптеке. Обратная ситуация: видит в аптеке ЧГ
незнакомый Brufen и не знает, что это тот же ибупрофен.

Идея: связать действующее вещество (INN) с торговыми названиями в разных странах и
показывать это на карточке препарата:
> «Brufen — это ибупрофен. В России продаётся как Нурофен, в США — Advil/Motrin.»

## 2. Scope (сужен по итогам обсуждения)

**В scope:**
- Только **формы продажи** = торговые названия по странам (набор стран — см. §5.4).
- Только вещества, **популярные у туристов** (аптечка путешественника), не весь регистр.
  Курируемый список ~30–60 веществ, а не все 904.
- Черногорская сторона берётся из существующих `med_medicines` (ничего досбора не надо).
- Подсказка «в Черногории нет лекарства с таким веществом», если оно у нас не
  зарегистрировано (кейс Энтеросгеля, Мирамистина и т.п.).

**Вне scope (сознательно):**
- Регуляторный статус «разрешено / не разрешено» в стране — НЕ собираем (волатильно,
  дорого, юридически скользко).
- «Вещества этой группы» / межгрупповая взаимозаменяемость — НЕ показываем как
  альтернативу (YMYL-риск, см. §6).
- Дозировки/схемы приёма зарубежных брендов — НЕ переносим (разные формы/дозы по странам).

## 3. Выводы по SEO

Положительно, при подаче через INN + реальная польза на странице:
- Длинный хвост с высоким интентом: «аналог нурофена в Черногории», «чем заменить X»,
  «нурофен по-черногорски». Низкая конкуренция, аудитория уже в стране.
- Усиление топического кластера по фарме + внутренняя перелинковка
  (вещество ↔ бренды ↔ препараты ЧГ).
- Зарубежные бренды становятся поисковыми синонимами к нашим карточкам.

Главный риск — **scaled content abuse** (helpful-content / март-2024). Поэтому:
- Фаза 1 НЕ создаёт новых страниц — только обогащает существующие карточки. Нулевой
  риск тонкого контента.
- Отдельные лендинги под бренд-запросы — только если Фаза 1 покажет спрос, и только с
  уникальным содержимым (§7, Фаза 3).

**Таб vs sitemap (основная цель — SEO):** проверено в коде — играет нам на руку.
- «Табы» в приложении (`components/entity-page/tab-bar.vue`) — это **scroll-spy по якорям**,
  а не скрытые панели: кнопка плавно скроллит к секции и пишет `?tab=id` в URL, но **весь
  контент секций всегда отрендерен в SSR-DOM** (`v-if`/догрузки по клику нет). Значит контент
  таба индексируется без спецусилий — это лучший вариант для SEO.
- Таб **не создаёт новый URL** (`?tab=` — квери-параметр на той же странице; карточка
  препарата уже в sitemap, генерится в `server/common/sitemap/sitemap.ts:223-231`). Якоря/
  квери в sitemap не добавляют. Так что «само» — да, отдельного действия по sitemap нет.
- Названия зарубежных брендов (Нурофен, Advil) — **текстом в HTML** → карточка ранжируется
  по «нурофен черногория» и т.п. Плюс `alternateName` в schema.org Drug (`activeIngredient`
  там уже есть; §7, задача 5). Это и есть SEO-механика фичи; лендинги (Фаза 3) — потом.

## 4. Выводы по YMYL

Концепция допустима, но опасна конкретная формулировка. Правила подачи:
- **Якорь на действующее вещество (INN), не «бренд = бренд».**
  Не «Brufen = Нурофен», а «Brufen содержит ибупрофен; в РФ ибупрофен продаётся как
  Нурофен». Бренд-к-бренду выводится как следствие, но семантически это «то же вещество».
- **Информационная рамка, не рекомендательная.** Никаких «замените X на Y». Дисклеймер
  на каждом блоке: справка, не назначение; уточните у фармацевта/врача.
- **Разделять уровни строго:** «то же вещество» (показываем) vs «та же группа»
  (в scope НЕ входит).
- **Источники = E-E-A-T + защита.** Ссылка на WHO ATC/INN + официальные регистры.
  Официальный источник усиливает авторитетность страницы и снимает часть ответственности.

Комбинированные препараты (Терафлю, Пенталгин, Корвалол) — отдельный аккуратный случай:
у них несколько веществ, «прямого аналога» часто нет. Для таких либо не показываем
связку, либо честно пишем «комбинированный препарат, прямого аналога нет».

## 5. Данные

### 5.1. Что уже есть (ЧГ-сторона, System A)
`med_substances` (INN, переводы 6 языков) ← `med_medicine_substances` → `med_medicines`
(бренды ЧГ). То есть по substance_id мы уже умеем достать список черногорских брендов.

### 5.2. Ключевой вывод из анализа схемы (влияет на модель)
`med_substances` пишется идемпотентным `INSERT ... ON DUPLICATE KEY UPDATE` по
натуральному ключу `name` (UNIQUE, prefix 400). Скрейпер CInMED **никогда** не делает
TRUNCATE/DELETE, не затирает `name`/`id` чужих строк и не трогает новые колонки. Связи
`med_medicine_substances` резолвятся **по имени вещества** (`JOIN ... ON s.name = '<sub>'`),
а не по хардкодным id. Следствия:
- Безопасно **добавлять наши курируемые вещества прямо в `med_substances`** (ключ — `name`).
- Безопасно добавить новую колонку через `ALTER` (скрейпер её не пишет).
- Нельзя ссылаться на вещества по хардкодному `id` — только по `name` (id auto-increment).

### 5.3. Модель данных (пересмотрено — решение проблемы дубля)
Проблема исходного варианта: `substance_id = NULL` для «нет в ЧГ» → когда препарат
приедет в ЧГ, в `med_substances` появится строка, а в справочнике брендов останется
висячая NULL-строка с тем же INN = **дубль**.

Решаем через единую таблицу веществ (как ты и предложил), без второй сущности:

1. **Каждое** туристическое вещество = строка в `med_substances`. Большинство (ибупрофен,
   парацетамол…) уже там из CInMED; недостающие (Энтеросгель, Мирамистин) добавляем
   вручную одной идемпотентной вставкой, ключ — `name`.
2. `med_foreign_brands.substance_id` теперь **NOT NULL** — висячих строк не бывает.
3. «Есть/нет в ЧГ» — **не храним флагом, а выводим**: существует ли активный
   `med_medicines`, связанный с веществом через `med_medicine_substances`. Когда препарат
   приедет в ЧГ, скрейпер сам добавит medicine + связку (матчинг по `name` вещества) →
   «нет в ЧГ» автоматически станет «есть». Ручной флаг «одобрен/не одобрен» протух бы —
   его никто не переключит в момент прихода препарата.

Хранение (решено): **отдельная таблица, НЕ строки в `med_medicines`**. Причина — `med_medicines`
= «продаётся в ЧГ», по ней бьют списки/фильтры/счётчики/sitemap/поиск/аналоги; подмешивание
иностранных потребовало бы `WHERE market='ME'` в каждом запросе + коллизии по `cinmed_id`.
Изоляция важнее переиспользования. «Таблицы разрешений» нет (регстатус не собираем);
единственная привязка — `substance_id`.

Гранулярность (решено): **связь на уровне вещества, дозировка/форма — описанием** (не
«продукт=продукт»). Поэтому `strength`/`pharma_form` — необязательные описательные поля,
а не ключ матчинга. Локальная карточка свою дозу показывает сама.

```sql
-- миграция NNN-med-foreign-brands.sql
CREATE TABLE med_foreign_brands (
  id            INT AUTO_INCREMENT PRIMARY KEY,
  substance_id  SMALLINT UNSIGNED NOT NULL,   -- FK med_substances.id (тип совпадает!)
  market_code   VARCHAR(4)   NOT NULL,        -- RU / DE / PL / US (рынок, см. §5.4)
  brand_name    VARCHAR(255) NOT NULL,        -- торговое название (Нурофен)
  strength      VARCHAR(120) NULL,            -- дозировки как текст-справка («200/400 мг»)
  pharma_form   VARCHAR(120) NULL,            -- форма как текст-справка («таблетки, сироп»)
  note          VARCHAR(500) NULL,            -- напр. «в США безрецептурный только 200 мг»
  created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  UNIQUE KEY uq_brand (substance_id, market_code, brand_name),
  KEY idx_market (market_code),
  CONSTRAINT fk_fb_substance FOREIGN KEY (substance_id)
    REFERENCES med_substances(id) ON DELETE CASCADE ON UPDATE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

Замечания:
- `strength`/`pharma_form` — свободный текст (справка), НЕ структура для дозо-матчинга.
- `inn_en`/`atc_code` из справочника брендов убраны — INN берётся из `med_substances`
  (`name_en`), бренды всегда через `substance_id`.
- ЧГ-бренды НЕ дублируем — берём из `med_medicines` по связке.
- SET NAMES utf8mb4 COLLATE utf8mb4_unicode_ci в импортных SQL (наш стандартный gotcha).

**Остаточный (мелкий) риск дубля.** У ручного вещества `name` надо задать той же
черногорско-сербской строкой INN, которую использовал бы CInMED, — тогда будущий
авто-JOIN сработает и препарат подцепится. Если написание не угадаем, CInMED позже
вставит вторую строку с иным `name` → дубль. При ~40 курируемых веществах это ловится
руками: на каждом обновлении CInMED сверять ~5 «нет в ЧГ» веществ. Не проблема масштаба.

### 5.4. Страны (уточнено по статистике турпотока 2025)
Данные Monstat 2025: Сербия #1 (~16%), Босния (~5%), Германия (~5–6.5%), Франция (~4.5%),
Польша (~4.5%), UK (~7% в сент.), Израиль (~9%), США (~3%). Ключевой фильтр — **разный
фармрынок**: у Сербии и Боснии бренды те же, что в ЧГ (общий экс-югославский рынок), их
турист и так видит «свою» карточку → **не собираем**.

`market_code` — это **рынок**, не строго страна ISO (бренды общие для группы стран):

- **RU** — «Русскоязычные страны» (Россия, Украина, Казахстан). Общий русскоязычный
  фармрынок с одинаковыми брендами (Нурофен, Но-шпа, Смекта). Хранится одной группой,
  чтобы не троить идентичные данные. Ярлык в UI — «Россия, Украина, Казахстан», **не**
  «СНГ» и **не** «пост-СССР» (Прибалтика в этот рынок не входит — она на рынке ЕС).
- **UA** — заводится **отдельным** рынком, только если на этапе сбора обнаружится много
  различий (украиноязычные написания, иные регистрации). По умолчанию — внутри RU.
- **DE** — Германия, крупнейший западный поток с иными брендами.
- **PL** — Польша (~4.5% потока, подтверждает твою интуицию).
- **US** — английские бренды (Advil/Tylenol) глобально узнаваемы + легче всего собрать
  из открытых источников.
- (опционально, если сбор дёшев) **UK**, **FR**.

Сербию/Боснию сознательно исключаем — тот же рынок, что у ЧГ. EU одной прокси-страной
**не** покрываем: берём конкретно DE и PL.

## 6. UX на карточке (System A, `pages/medicines/[medicineSlug]/index.vue`)

Отдельный **таб «Аналоги в других странах»**. По факту паттерна (см. §3) это =
новая `EntityPageSection sectionId="foreign"` в слоте `#sections` +
запись `{ id: 'foreign', label }` в computed `tabs`
(`pages/medicines/[medicineSlug]/index.vue:201-207`). Контент уедет в SSR-DOM
автоматически, таб-бар лишь скроллит к нему. Скрывать/`v-show` ничего не нужно.

Содержимое секции:

> **Аналоги в других странах**
> Действующее вещество — **ибупрофен**.
> 🇷🇺 Россия/Украина/Казахстан: Нурофен (табл. 200/400 мг, сироп)
> 🇩🇪 Германия: Nurofen, Ibuprofen (200/400 мг) · 🇵🇱 Польша: Ibuprom
> 🇺🇸 США: Advil, Motrin (200 мг)
> _Справочная информация. Формы, дозировки и состав в разных странах могут отличаться —
> уточняйте у фармацевта._

Подача — **на уровне вещества** (не «продукт=продукт»): бренды перечислены с их
дозировками/формами как справка; свою дозу локальный препарат показывает в основном блоке.

Кейс «нет в ЧГ» (у вещества нет активных связанных `med_medicines`) — на будущей
странице вещества или в поиске: «Препаратов с этим веществом в аптеках Черногории нет.»

Поиск: зарубежные бренды индексируются как синонимы → запрос «нурофен» ведёт к
веществу ибупрофен → к черногорским брендам. (Интеграция в существующий поиск по
`/medicines`, отдельная задача.)

- **Страны** → рынки RU (рус.-язычные: RU/UA/KZ), DE, PL, US; UA отдельно только при
  больших различиях; Сербию/Боснию не берём (общий рынок с ЧГ). (§5.4)
- **«Нет в ЧГ»** → выводим из отсутствия активных `med_medicines`, не храним флагом. (§5.3)
- **Дубль при приходе препарата в ЧГ** → единая таблица `med_substances`, `substance_id`
  NOT NULL; остаточный риск (написание `name`) ловится вручную при ~40–100 веществах. (§5.3)
- **Комбинированные (Терафлю/Пенталгин/Корвалол)** → **не в 1-й итерации**. Бренд цепляется
  к одному веществу. Идея на итерацию 2: матчить зарубежную комбинацию на местный
  многокомпонентный препарат ЧГ по совпадению набора веществ (может совпасть, может нет) —
  переиспользовать существующую set-matching логику аналогов. Сейчас — только однокомпонентные.
- **Состав аптечки туриста** → ростер собирает Claude (~97 позиций), утверждён Anton.
  Supplement-позиции (валериана, пустырник=maričnjak, витамины) — в отдельную итерацию
  (CInMED БАДы не выверяет, данных мало).
- **Хранение** → отдельная таблица `med_foreign_brands`, НЕ строки в `med_medicines`
  (изоляция от ЧГ-регистра). (§5.3)
- **Дозировки/гранулярность** → связь на уровне вещества; `strength`/`pharma_form` —
  описательная справка, не ключ дозо-матчинга «продукт=продукт». (§5.3, §6)
