## Цель

Полностью убрать Element Plus (`element-plus`, `@element-plus/nuxt`, транзитивный
`@element-plus/icons-vue`) из проекта, заменив его компактным собственным UI-китом,
построенным на существующей системе дизайн-токенов.

## Контекст: почему вопрос вообще встал

Проект фактически уже не использует Element Plus как дизайн-систему — только как
библиотеку контролов. Дизайн-решения живут в собственных токенах
(`assets/css/design-tokens.css`), а EP подгоняется под них через мост `--el-*`.
История проекта — это история постепенного выдавливания EP:

- собственная система токенов + мост `:root:root { --el-* }`;
- кастомный `AppDialog` (Teleport, `inert`, блокировка скролла) — EP-диалог уже не нужен;
- `el-breadcrumb` запрещён конвенцией, везде `AppBreadcrumbs`;
- точечные бои с дефолтами EP: база 16px против EP 14px (авто-зум iOS),
  затемнение danger-кнопок ради контраста 4.5:1, переопределение теней/радиусов;
- Airbnb-типографика на карточках — мимо EP полностью.

Каждая новая фича начинается с «как заставить EP выглядеть не как EP».

## Ценность

| Выгода | Для кого | Комментарий |
|---|---|---|
| Меньший JS/CSS бандл на SEO-страницах | Пользователи, SEO | Каталоги (/clinics, /doctors, /services, /medications, /medicines, /labtests) — органический трафик, скорость = деньги. EP-рантайм и стили уходят из публичного контура целиком |
| Полный контроль дизайна | Продукт | Конец мосту `--el-*` и переопределениям; вид компонентов определяется только токенами |
| Тёмная тема дешевле | Roadmap | Свой кит = второй набор токенов. С EP — поддержка двух систем переменных |
| Доступность под контролем | Пользователи | WCAG-фиксы перестают быть борьбой со стилями EP |
| Одна скрытая зависимость меньше | Разработка | `@element-plus/icons-vue` импортируется в 27 файлах, но не объявлен в package.json — работает только потому, что хостится транзитивно |

## Оценка идеи (честная)

### Аргументы «за»

1. **Траектория уже задана.** Половина пути пройдена (токены, диалоги, breadcrumbs,
   карточки). Текущее состояние — худшее из обоих миров: и зависимость тянем,
   и дизайн всё равно свой.
2. **Профиль использования узкий.** Из ~70 компонентов EP реально используется ~25,
   из них 6 дают 80% вхождений (button, input, select, icon, form, tag).
   Кит из ~15 компонентов закрывает всё.
3. **Сложные виджеты EP сидят в непубличной зоне.** table, tabs, time-picker —
   только админка и кабинет (`ssr: false`). Публичному контуру нужны простые вещи.
4. **Сайт consumer/mobile-first, EP — desktop/admin-ориентированный.** Селекты EP
   на мобильных каталогах — постоянный источник UX-компромиссов.

### Аргументы «против» (и почему они не блокируют)

1. **Объём: ~95 файлов, ~500 вхождений.** Реально, но 140 из них — `el-button`
   (тривиальная замена), а тяжёлая часть сконцентрирована в админке, которую можно
   мигрировать последней или отложить.
2. **Ноль новых фич для пользователя.** Правда. Поэтому приоритет P2 и миграция
   контурами: публичный контур окупается бандлом и развязывает руки тёмной теме,
   остальное — по мере свободного ресурса.
3. **Доступный select/tooltip с нуля — это дорого.** Самый серьёзный риск.
   Смягчение: (а) спискам проекта не нужна виртуализация (max ~130 элементов
   в админском поиске клиник — `el-select-v2` там был перестраховкой);
   (б) если полноценный combobox забуксует — точечный fallback на headless-библиотеку
   (Reka UI) только для listbox/combobox, это поведение без стилей, не дизайн-система.
4. **Риск визуальных/a11y регрессий.** Смягчение: кит строится на тех же токенах,
   на которые уже намаплен EP — визуальный дифф минимален; e2e Playwright +
   повторный WCAG-проход по новым компонентам.

### Рассмотренные альтернативы

| Вариант | Вердикт |
|---|---|
| Оставить как есть (EP + мост) | Постоянный налог на каждую фичу; тёмная тема удваивает мост; скрытая зависимость иконок остаётся |
| Перейти на другую библиотеку (PrimeVue, Naive, Vuetify) | Та же проблема в другой обёртке: чужие дефолты против своих токенов |
| Headless-библиотека (Reka UI) как база всего кита | Рабочий вариант, но проект уже доказал способность писать свои примитивы (AppDialog с inert). Берём как точечный fallback для combobox, не как фундамент |
| Свой кит на своих токенах | **Выбран.** Минимум зависимостей, полный контроль, прямой путь к тёмной теме |

### Вердикт

**Идея здравая, делать стоит — но строго фазами и без «большого взрыва».**
Ценность сконцентрирована в публичном SSR-контуре (итерации 1–3) — он самодостаточен:
даже если миграция остановится после итерации 3, проект получит быстрый публичный
сайт на своём ките, а EP останется изолированным в `ssr: false` зоне. Полное
удаление пакета (итерация 5) — желательная, но не обязательная конечная точка
с фиксированным дедлайном.

## Ограничения

- Никаких новых runtime-зависимостей по умолчанию (исключение — осознанное решение
  по combobox, см. риски).
- Визуальный паритет «по умолчанию»: компонент кита на старте выглядит так же, как
  EP-аналог под текущим мостом токенов. Редизайн — отдельные решения, не часть миграции.
- Тексты и aria-атрибуты — только через props/i18n, без зашитых строк
  (конвенция i18n-аудита 2026-06).
