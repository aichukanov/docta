## Краткое описание

Поэтапная замена Element Plus собственным UI-китом на базе уже существующей
системы дизайн-токенов (`assets/css/design-tokens.css`). Цель — полный контроль
над дизайном, меньший клиентский бандл на SEO-страницах и снятие постоянных
конфликтов с дефолтами EP (типографика 14px, контраст, стили контролов).

**Ключевой принцип:** миграция идёт контурами — сначала публичные SSR-страницы
(максимум пользы), затем auth/профиль/кабинет, админка — последней. Каждая
итерация заканчивается «grep-гейтом»: в закрытом контуре не остаётся ни одного
`el-*` / `ElMessage` / `@element-plus`.

### Что уже сделано до этого PRD (стартовая позиция)

- Своя система дизайн-токенов + мост `--el-*` поверх неё (`:root:root`)
- Кастомные `AppDialog` (Teleport, inert, focus-safe), `ConfirmDialog`, `AppBreadcrumbs`, `rating-stars`
- Конвенция «переиспользовать общие компоненты вместо EP-аналогов» (el-breadcrumb запрещён)
- Обёртки-точки замены: `pagination.vue` (el-pagination), `filterable-select.vue` (el-select-v2)

### Зависимости и связи

- **Тёмная тема** (roadmap из code review 2026-06) — делать ПОСЛЕ этой миграции: на своём ките это просто второй набор токенов, с EP пришлось бы дублировать мост под dark-переменные EP
- **WCAG-аудит 2026-06** — закрыт на текущем UI; новые компоненты обязаны держать ту же планку (см. NFR)
- Playwright e2e — основная сетка регрессионных проверок миграции

## Технологии

- **UI-кит:** свои Vue 3 SFC в `components/ui/`, стили только на токенах
- **Иконки:** локальные SVG-компоненты (копия используемых иконок EP, MIT) — снимает скрытую транзитивную зависимость `@element-plus/icons-vue`
- **Валидация форм:** лёгкий composable + существующие правила, без аналога async-validator
- **Уведомления:** `useToast()` composable + хост-компонент вместо `ElMessage`
