# Задача: довести аналитику до конца (типизированные события взаимодействий)

Реализуй оставшуюся кодовую часть PRD `prd/analytics/` — итерации 2 (остаток),
3 (события взаимодействий) и кодовую часть 4 (свойства событий под воронки).

## Перед началом прочитай

- `prd/analytics/02-requirements.md`, `prd/analytics/03-architecture.md`
- `prd/analytics/iterations/iteration-02-page-views.md`, `iteration-03-interactions.md`,
  `iteration-04-funnels.md`
- Текущий код: `composables/use-analytics.ts`, `components/cookie-modal.vue`,
  `composables/use-cookie-control.ts`, `components/contacts/copy-button.vue`
  (единственное существующее событие `contact_copied`)

## Важные поправки к PRD (документация устарела)

- Итерация 1 уже сделана: Mixpanel инициализируется динамическим импортом после
  consent, `track_pageview: true` уже автотрекает просмотры страниц. Плюс работают
  GA4 (gtag) и Cloudflare Insights. НЕ переделывай инициализацию.
- Auth полностью реализован — `identifyUser` уже есть в use-analytics, но нигде
  не вызывается: подключи к логину/логауту (`server/api/auth/active-user.ts`,
  стор пользователя — найди, где фронт узнаёт о логине).
- Избранного (favorites) в проекте НЕТ — события избранного из PRD пропусти.
- Unit-тестов в проекте нет (только Playwright e2e) — НЕ вводи новый тестовый
  фреймворк, ограничься typecheck и ручной проверкой через mixpanel debug mode.

## Что сделать

1. **Каталог типизированных событий** — `types/analytics.ts`: union имён событий +
   интерфейсы пропсов для каждого. `trackEvent` в use-analytics сделай дженериком
   по этому каталогу, существующий вызов `contact_copied` приведи к типам.
   Снейк-кейс имён, у каждого события — `entity_type` ('clinic' | 'doctor' |
   'service' | 'labtest' | 'medicine' | 'article'), `entity_id`, `entity_slug`,
   чтобы воронки итерации 4 строились без переименований.
2. **Super properties**: locale, тип страницы — региструй при инициализации.
3. **События взаимодействий** (итерация 3, по списку в PRD):
   - клик по телефону (tel: ссылки), копирование телефона (есть), клик/копирование
     email, копирование адреса;
   - переход на сайт клиники (внешняя ссылка);
   - переходы между сущностями: service→clinic, doctor→clinic, clinic→doctor,
     medicine→clinic (карточки и кнопки на детальных страницах);
   - использование поиска и фильтров на листингах (debounce, трекай применённый
     фильтр и его значение, НЕ каждый кейстрок);
   - взаимодействие с картой (открытие карты, клик по маркеру) — координируйся
     с зоной окна 1, см. ограничение ниже.
4. **User identification**: identify при логине, reset при логауте, user property —
   способ авторизации (google/facebook/telegram/password).
5. **Документация**: каталог событий с описанием и свойствами — в
   `prd/analytics/events-catalog.md`. Инструкцию по настройке воронок/дашбордов
   в UI Mixpanel (итерация 4 — это конфигурация, не код) опиши там же как
   пошаговый чеклист для пользователя.

## Порядок фаз (важно — параллельное окно)

Окно 1 параллельно переделывает `pages/clinics/index.vue`, `server/api/clinics/list.ts`
и карточки клиник (геолокация, карта, фильтры).

- Фаза А (делай сразу): типы, use-analytics, identify, детальные страницы врача/
  клиники/услуги/анализа/лекарства, компоненты контактов.
- Фаза Б (строго последней, перед ней спроси пользователя, закончило ли окно 1):
  трекинг фильтров/поиска на листингах и события карты.

## Конвенции проекта (обязательно)

- Никаких изменений в consent-логике без необходимости; события не должны
  отправляться без согласия (проверка уже есть в trackEvent — сохрани её).
- В события не класть PII: телефон/email в пропсах события заменяй на
  entity_id + тип контакта (существующий `contact_copied` шлёт value — исправь).
- `npm run typecheck` зелёный. НЕ запускай `nuxt build` при работающем dev-сервере.
- НЕ коммить — пользователь коммитит сам.

## Definition of Done

- Каталог событий покрывает итерации 2–3 PRD (минус избранное), всё типизировано.
- identify/reset подключены к auth-флоу.
- `prd/analytics/PROGRESS.md` и итерации обновлены под реальность (итерация 1 —
  done ранее, что именно сделано сейчас, что осталось пользователю в UI Mixpanel).
- Итоговая сводка: список событий, изменённые файлы, как проверить в debug mode.
