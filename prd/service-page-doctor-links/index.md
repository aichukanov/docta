# Врачи в карточке клиники на странице услуги

## Проблема

Карточки врачей (`/doctors/[slug]`) структурно самый слабо-перелинкованный тип сущности на сайте. Подтверждено вживую через GSC URL Inspection (2026-07): у карточки врача с ровно 1 входящей ссылкой (со страницы штата своей же клиники) статус «Crawled — currently not indexed», тогда как у карточки услуги с 5 входящими ссылками (языковые альтернативы + листинг) — «проиндексировано». При低ком трасте домена (см. [[seo-audit-2026-07]]) решает не объём текста на странице, а количество и разнообразие внутренних ссылок.

Дополнительно: 1110 из 1311 опубликованных врачей (85%) не имеют `description` ни на одном языке — контентное решение (автогенерируемый факт-блок) отклонено пользователем 2026-07-23: будет дублированием структурных полей, а `description` заполняется вручную врачом/админом. Если позже появятся новые поля (образование, научные работы) — они пойдут в отдельный таб, не в `description`. Эта PRD решает только сторону перелинковки.

**Решение:** на странице услуги, у каждой клиники в списке (там уже есть цена — `EntityPageClinicsSection`/`ClinicSummary`), показать 1-2 врачей, работающих в этой клинике по нужному профилю. Новый источник входящих ссылок на карточки врачей — потенциально ~3216 услуг из 5228.

## Scope

**В скоупе:** только `medical_services` (страница `/services/[serviceSlug]`).

**Не в скоупе — `lab_tests` (анализы).** В схеме БД нет `medical_services_specialties`-аналога для анализов и нет `clinic_lab_test_doctors` — анализы не привязаны к специальности/врачу так, как процедуры (анализ обычно назначает/делает любой врач или сама лаборатория). Не пытаться притягивать эту фичу к `/labtests/[labTestSlug]` без отдельного решения о данных.

**Не в скоупе:** обратная сторона (список услуг на карточке врача) — не нужна для решения проблемы перелинковки врачей, отдельная задача при необходимости.

**Не в скоупе:** контентные поля у врача (образование, публикации) — уже отклонено пользователем, только структурные ссылки.

## Источники данных (проверено на локальной БД 2026-07-23)

Два источника, приоритет — точный:

### 1. Точная связь (приоритет)

Таблица `clinic_medical_service_doctors` (уже существует, наполняется клиникой через кабинет — см. [[clinic-cabinet-billing]]):

```sql
CREATE TABLE clinic_medical_service_doctors (
  id INT PRIMARY KEY AUTO_INCREMENT,
  clinic_id INT,
  medical_service_id INT,
  doctor_id INT,
  created_at DATETIME,
  price DECIMAL(10,2),
  price_max DECIMAL(10,2)
);
```

Текущее наполнение: **159 строк, 78 услуг (1.5% из 5228), 37 врачей (2.8% из 1311)**. Точная, подтверждённая клиникой связь «этот врач оказывает эту услугу в этой клинике» — но покрытие маленькое, само по себе не решает проблему тонкой перелинковки в масштабе.

### 2. Фолбэк — по специальности (когда для данной пары клиника+услуга нет точных строк)

```sql
medical_services_specialties (medical_service_id, specialty_id)  -- 4353 строк, 3875 услуг охвачено (74%)
doctor_specialties (doctor_id, specialty_id)                      -- покрывает всех 1311 врачей
doctor_clinics (doctor_id, clinic_id)                             -- 1860 строк, все 1311 врачей, 127 клиник
```

Join этих трёх таблиц с `clinic_medical_services` даёт врачей нужной специальности, работающих именно в клинике, которая показана на странице услуги (**не сайтвайд** — иначе задвоит уже существующий хаб специальность×город из `components/doctor/related-filters.vue`). Проверенное покрытие: **3216 из 5228 услуг (61.5%)** получат хотя бы одного врача через этот join.

Пример запроса, который уже проверен на реальных данных и даёт от 4 до 19+ подходящих врачей на услугу (нужен кэп, см. ниже):

```sql
SELECT dc.clinic_id, d.id AS doctor_id, d.rank_score
FROM clinic_medical_services cms
JOIN doctor_clinics dc ON dc.clinic_id = cms.clinic_id
JOIN doctor_specialties ds ON ds.doctor_id = dc.doctor_id
JOIN medical_services_specialties mss
  ON mss.specialty_id = ds.specialty_id AND mss.medical_service_id = cms.medical_service_id
JOIN doctors d ON d.id = dc.doctor_id AND d.hidden = 0 AND d.is_draft = 0
WHERE cms.medical_service_id = ?
  AND cms.clinic_id IN (<clinicIds, уже отобранные для страницы услуги>)
ORDER BY d.rank_score DESC;
```

### Слияние источников (за клинику)

Для каждого `clinic_id`, показанного на странице услуги:

1. Если в `clinic_medical_service_doctors` есть строки для этой пары (`medical_service_id`, `clinic_id`) — берём их (обычно 1-2, кэп не нужен, но поставить `LIMIT 2` на всякий случай).
2. Иначе — берём фолбэк по специальности, **кэп 2 врача**, сортировка `d.rank_score DESC` (колонка уже существует и используется для ранжирования в других местах — `getClinicRankOrderBySQL`).

## UI

**Важно про формулировку.** У точного источника подтверждено «делает эту процедуру», у фолбэка — только «специальность совпадает, работает в этой клинике». UI не различает источник визуально (одна и та же карточка), значит подпись блока обязана быть верной для обоих случаев одновременно. **Не писать** «Врачи, которые делают эту услугу» — только нейтральное **«Врачи»** / **«Specialists»** (заголовок без утверждения о личном выполнении процедуры). См. [[data-accuracy]].

### Место в разметке

Пользователь предложил структуру: **услуга → клиника (с ценой) → список врачей**, вложенный в существующую карточку клиники, а не отдельная секция/таб на странице услуги. Это совпадает с уже существующим компонентом:

- `components/clinic/summary.vue` — уже есть `el-collapse` с пунктами «Профильные услуги» (`services` prop, используется в контексте карточки клиники) и «Контакты». Добавить третий `el-collapse-item` **«Врачи»**, показывается только если `doctors.length > 0` (тот же паттерн, что `hasServices`/`hasClinicContacts`).
- Новый проп: `doctors?: DoctorCardData[]` (тип уже существует, `interfaces/doctor.ts`).
- **Не создавать новый компонент карточки врача.** `components/doctor/info.vue` уже поддерживает компактный режим (`short` проп) — рендерит аватар+имя-ссылку+специальности+рейтинг в сжатом виде, принимает `DoctorCardData`. Использовать `<DoctorInfo :service="doctor" short />` в цикле. Проверить вживую, не слишком ли шумно смотрится блок специальностей (`DoctorSpecialties` внутри `info.vue`) в таком сжатом контексте — если да, можно скрыть его именно в этом месте отдельным пропом, но это на усмотрение реализующего агента, не блокер.

## Acceptance Criteria

- AC-1: На странице услуги, у которой хотя бы одна клиника имеет точную или фолбэк-связь с врачом, в карточке этой клиники появляется блок «Врачи» с 1-2 именами, каждое — рабочая ссылка на `/doctors/[slug]`.
- AC-2: Если для клиники нет ни точных, ни фолбэк-врачей — блок «Врачи» не рендерится вообще (не пустой заголовок).
- AC-3: Если для пары клиника+услуга есть точные строки в `clinic_medical_service_doctors` — показываются именно они, фолбэк не подмешивается.
- AC-4: Фолбэк-список ограничен 2 врачами на клинику, сортировка по `rank_score DESC`.
- AC-5: Формулировка заголовка блока не утверждает, что врач лично выполняет процедуру (нейтральный лейбл, см. раздел UI).
- AC-6: Раздел не появляется на `/labtests/[labTestSlug]` — эта PRD туда не расширяется.
- AC-7: Проверено вживую на dev-сервере минимум на 3 услугах (одна с точными данными, одна только с фолбэком, одна без совпадений вообще) и во всех 6 локалях (имя врача и подпись блока локализуются).
- AC-8: `npm run typecheck` зелёный.

## Риски

- Фолбэк может дать врача, который физически в клинике уже не работает или в отпуске/неактивен — та же степень неточности, что и у остальных списков по специальности на сайте (хаб 1b), не выше. Если появятся жалобы от клиник — рассмотреть флаг `doctor_clinics.active` (сейчас такого поля нет, не в скоупе этой PRD).
- Не заводить кэш/материализацию — объём джойнов небольшой (несколько тысяч строк на таблицах), обычный запрос на каждый рендер страницы, как и остальные секции `services/details.ts`.
