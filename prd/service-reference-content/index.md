# Справочный контент на страницах услуг и анализов

## Проблема

Страницы `pages/services/[serviceSlug]/index.vue` и `pages/labtests/[labTestSlug]/index.vue` содержат только вкладки «Клиники / Тариф ФЗОЦГ / Карта» — текстового контента нет. Это часть причины thin-content потолка сайта (см. [[seo-audit-2026-07]], пункт 4 плана).

## Формат справки (единый для всех страниц)

Пять полей, без воды, таблицей:

| Поле | Смысл | Ограничение по формулировке |
|---|---|---|
| **Что это** (`what`) | 1 фраза, что за анализ/процедура | Без диагностических ярлыков |
| **Как проводится** (`how`) | Что физически происходит, сколько занимает | Только процедура, не результат |
| **Когда назначают** (`when`) | Типичные показания | Общие ситуации, не «у вас есть X» |
| **Подготовка** (`prep`) | Условия перед процедурой | Конкретно и проверяемо (натощак, часы) |
| **О чём может говорить отклонение** (`abnormal`) | Направление, не диагноз | Обязательно «может быть связано с» + отсылка к врачу |

Плюс единый дисклеймер на каждой странице (не дублируется в данных — общая i18n-строка):

> Информация носит справочный характер и не заменяет консультацию врача. Референсные значения и трактовка результата зависят от лаборатории/клиники и индивидуальных особенностей пациента.

### YMYL-правила

- Никаких числовых референсных диапазонов (разные у лабораторий/приборов).
- Никаких формулировок «повышение X означает болезнь Y» — только «может быть связано с» + отсылка к врачу.
- Никаких рекомендаций по лечению/дозировкам.
- Онкомаркеры/ВИЧ/гепатиты/сифилис — только нейтральный факт-контент (что это, зачем берут), без интерпретации риска и без стигматизирующих формулировок.
- Пренатальный скрининг с расчётом риска (Double/Triple test, PAPP-A, free-beta-hCG) — по умолчанию вне охвата, слишком сложна нюансная интерпретация для нейтрального текста. Исключение (см. батч 7 в PROGRESS.md, 2026-07-18): если конкретная позиция уже нужна под статью, карточку пишем, но без расчёта/интерпретации риска — поле `abnormal` ограничивается «оценивается врачом только в комплексе с остальными показателями скрининга», без упоминания трисомий/вероятностей.
- Тесты на наркотики — вне охвата (не медицинский чекап по сути, юридический контекст).

## Хранение

**Черновик (текущее состояние):** JSON-файлы, переводы внутри.

Файлы:
- `data/entity-reference/lab-tests.json`
- `data/entity-reference/medical-services.json`
- `data/entity-reference/README.md` — схема

Схема на 1 элемент:

```json
{
  "slug": "complete-urinalysis",
  "entity_type": "lab_test",
  "translations": {
    "ru": {
      "what": "...",
      "how": "...",
      "when": "...",
      "prep": "...",
      "abnormal": "..."
    }
  }
}
```

Все 110 карточек переведены на все 6 локалей (`ru/en/sr/sr_cyrl/de/tr`) — см. PROGRESS.md.

**Финальная схема БД (решено 2026-07-17):** две отдельные 1:1-таблицы с настоящим FK — `lab_test_reference_info` (FK `lab_test_id` → `lab_tests.id`) и `medical_service_reference_info` (FK `medical_service_id` → `medical_services.id`). Не общая `entity_type`+`entity_id` таблица (как `review_ai_summaries`/`slug_redirects`) — предпочли ссылочную целостность дублированию структуры между таблицами.

Колонки — по конвенции суффиксов `name_en/name_sr/name_sr_cyrl/name_ru/name_de/name_tr` из `lab_tests`/`medical_services`, но по одной на каждое из 5 полей: `what_*`, `how_*`, `indications_*` (не `when_*` — зарезервированное слово в SQL), `prep_*`, `abnormal_*`. Итого 30 текстовых колонок на таблицу, все NULL (перевод может появляться по локали постепенно).

Миграция схемы: `server/sql/migrations/009-entity-reference-info.sql` — применена пользователем 2026-07-18. Импорт данных: `scripts/entity-reference/build-entity-reference-sql.mjs` → `server/sql/migrations/insert-entity-reference-info.sql` — применён (50 + 60 строк).

## Вывод на странице (реализовано 2026-07-18)

Важная деталь реализации: «табы» в `EntityPage` (`components/entity-page/tab-bar.vue`) — это не переключение видимости, а якорная навигация со скроллом. Все `EntityPageSection` в `#sections` рендерятся в DOM всегда, независимо от «активной» вкладки. Поэтому дилемма «таб vs блок для индексации» неактуальна — весь контент одинаково индексируется в обоих случаях.

Справка — `EntityPageSection`, первая в списке `#sections` (перед «Клиниками»), со своей записью в `tabs` (`TabReference`). Реализация: `server/common/utils.ts` (`buildReferenceInfo`), `server/api/{services,labtests}/details.ts`, `interfaces/reference-info.ts`, `components/entity-page/reference-section.vue`, обе страницы услуги/анализа. Проверено вживую на dev-сервере во всех 6 локалях.
