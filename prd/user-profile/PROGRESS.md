## Фактическая реализация (расходится с PRD-планом)

PRD писался до реализации auth; код ушёл дальше плана. Истина — в коде.

### Итерации 1–2 (сделаны ранее, шире PRD)

- Страница `/profile` с вкладками: профиль, безопасность, профиль врача, мои клиники
  (`pages/profile.vue` + `pages/profile/*.vue`, `components/profile/**`).
- Редактирование имени, смена email с подтверждением, смена пароля,
  загрузка собственного аватара (PRD предполагал только фото из OAuth).
- OAuth: Google + Telegram + Facebook (PRD планировал два), выбор primary-провайдера,
  отвязка (`server/api/auth/unlink/[provider].post.ts`), профили провайдеров
  в отдельных таблицах `auth_oauth_profiles_*`.
- Сессии и история входов (вкладка «Безопасность») — сверх PRD.
- Pinia store: общий `stores/user.ts` (`useUserStore`), отдельный userProfileStore не создавался.

### Итерация 3 — настройки приватности (2026-06-11)

- Поле `auth_users.is_profile_public BOOLEAN NOT NULL DEFAULT FALSE`
  (миграция `server/sql/migrations/004-user-privacy.sql`; поля `display_name` и
  `updated_profile_at` из PRD не нужны — имя уже редактируется в `auth_users.name`).
  Индекс по флагу из PRD не создавался: выборки всегда идут по PK/JOIN, индекс по
  boolean бесполезен.
- **Дефолт — приватный** (поправка из `prd/prompts/06`, PRD предлагал TRUE):
  реальные пользователи анонимны, пока сами не включат публичность.
  Исключение — фантомные пользователи (импортированные отзывы): в миграции
  бэкфилл TRUE + в логике отображения фантом всегда публичен, иначе все
  импортированные отзывы Google Maps/Facebook стали бы анонимными. Флаг
  переживает клейм фантома (континуитет подписи после первого входа).
- API: `POST /api/auth/update-privacy` (`{ isPublic: boolean }`), сохранение
  мгновенно при переключении, toast (FR-16). Отдельный GET не нужен — флаг входит
  в сессионного пользователя (`server/utils/session.ts` → `useUserStore`).
- UI: секция «Приватность» во вкладке профиля (`components/profile/tab/basic.vue`):
  el-switch публичный/приватный + живой предпросмотр подписи к отзыву
  (FR-21..24 реализованы инлайн-превью вместо модального окна — мгновенная
  обратная связь при переключении, меньше кода).
- Эффект: публичный — имя автора рядом с его отзывами; приватный — отзывы
  анонимны (без имени, фото и ссылки на профиль), включая schema.org-разметку
  (`components/reviews-page.vue` берёт автора из того же payload).

### Итерация 4 — маскировка email (2026-06-11)

- `common/email-masking.ts` → `maskEmail()`: `user@gmail.com` → `u*****@g****.*om`
  (реализация по эталонному коду из `03-architecture.md`; примеры в FR-20
  противоречили собственному коду PRD — за истину взят код).
- Применение: в `server/common/reviews.ts` подпись автора отзыва без имени —
  маскированный email вместо сырого (раньше сырой email утекал в публичный API
  через фолбэк `COALESCE(..., u.email)` — починено). Вспомогательные поля
  (`authorEmail`, `authorIsPublic` и пр.) вычищаются из ответа API.
- PRD-сценарий «маскированный email в комментариях» адаптирован: комментариев
  нет, авторство показывается в отзывах — маскировка применена там.
- Аудит публичных endpoint'ов на утечку email (по `prd/prompts/06`):
  `users/list.ts`, `reviews/list.ts`, `recent-from-real-users.ts` — requireAdmin;
  `test/email-preview` — только dev; `d.email`/клиничные email в details —
  публичные бизнес-контакты врачей/клиник, не email аккаунтов. Единственная
  утечка была в фолбэке подписи автора отзыва — устранена.
- Сверка критериев итераций 1–2 (по `prd/prompts/06`): FR-9/FR-10 (нельзя
  отвязать последний способ входа) — реализовано в
  `server/api/auth/unlink/[provider].post.ts`; остальные FR-1..FR-11 закрыты.
